# Как внести свой вклад в проект

Спасибо за ваш интерес к проекту! Этот документ описывает правила и рекомендации для разработчиков.

## 1. Регламент веток (GitFlow)

Мы используем модель ветвления GitFlow:

*   `main` — стабильная версия (Production). Прямые коммиты запрещены.
*   `develop` — ветка для разработки следующего релиза (Staging).
*   `feature/<issue-id>-<description>` — новая функциональность. Ветвится от `develop`.
    *   *Пример: `feature/42-add-queue-creation`*
*   `fix/<issue-id>-<description>` — исправление багов. Ветвится от `develop` (для плановых) или `main` (для хотфиксов).
*   `update/<issue-id>-<description>` — рефакторинг, обновление зависимостей.

## 2. Регламент коммитов

Каждый коммит должен соответствовать формату **Conventional Commits**. Это помогает автоматизировать генерацию changelog.

**Формат:** `<type>(<scope>): <subject>`

*   **type:** `feat` (новая фича), `fix` (багфикс), `docs` (документация), `style`, `refactor`, `test`, `chore` (обслуживание).
*   **scope (опционально):** часть проекта (api, db, auth, ci).
*   **subject:** краткое описание изменения.

**Примеры:**

feat(queues): add endpoint for queue creation

fix(auth): correct token expiration logic

docs(api): update response example for /users/me

## 3. Процесс работы над задачей

1.  Возьмите задачу из issue-трекера.
2.  Создайте ветку от `develop` по шаблону `feature/ISSUE-ID-description`.
3.  Реализуйте функциональность.
4.  Напишите или обновите тесты.
5.  Сделайте коммиты согласно регламенту.
6.  Откройте Pull Request (PR) в ветку `develop`.
7.  Убедитесь, что все проверки CI (линтеры, тесты) прошли успешно.
8.  Дождитесь одобрения (Code Review).
    *   **В `develop`:** требуется 1 approval.
    *   **В `main`:** требуется 2 approvals.
9.  После одобрения ваш PR будет смержен.

## 4. Настройка локального окружения

Все зависимости и сервисы (Go, React, PostgreSQL) запускаются в Docker. Просто следуйте инструкции в `README.md`.
Секреты для локальной разработки хранятся в файле `.env`, который не отслеживается Git. Используйте `.env.example` как шаблон.
