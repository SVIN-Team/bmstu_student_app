# Техническое задание проекта

### **1. Описание артефактов и терминов (Глоссарий)**

| Термин | Описание |
| :--- | :--- |
| **Система** | Разрабатываемое студенческое веб-приложение. |
| **Неавторизованный пользователь** | Пользователь, не прошедший регистрацию. |
| **Пользователь** | Общее название для любого лица, взаимодействующего с Системой. |
| **Студент** | Пользователь с ролью "Студент". Основной потребитель функциональности: просмотр расписания, запись в очереди |

| **Администратор** | Пользователь с максимальными правами. Управляет всеми данными в системе: аккаунтами пользователей, группами, расписанием, настройками системы. |
| **Учебная группа** | Группа, в которой обучается Студент (например, "ИУ7-81Б"). Используется для фильтрации расписания. |
| **Расписание** | Совокупность Занятий для учебных групп. Отображается в виде сетки по дням недели и времени. |

| **Очередь** | Электронный список для записи Студентов на сдачу лабораторной работы, экзамена или консультации. Создается Старостой. |

### **2. Продуктовое описание**

#### **2.1. Методология планирования**
При планировании функциональности проекта используется методология MoSCoW, которая позволяет классифицировать требования по степени их важности:
*   **Must Have** — критически важные функции, без которых система не может функционировать
*   **Should Have** — важные функции, которые желательно реализовать в текущей итерации
*   **Could Have** — полезные функции, которые можно реализовать при наличии времени и ресурсов
*   **Won't Have** — функции, которые не будут реализованы в текущей итерации, но могут быть рассмотрены в будущем

#### **2.2. Цели и задачи продукта**
Создать единое информационное пространство для студентов, которое позволит:
*   Удобно отслеживать актуальное расписание занятий.
*   Цифровизировать и упорядочить процесс записи на сдачу работ и экзаменов.
#### **2.3. Роли пользователей и их возможности**

**2.3.0. Гость**
*   Регистрация и авторизация в системе.

**2.3.1. Студент**
*   Просмотр и редактирование своего профиля (ФИО, группа). **[Must Have]**
*   Просмотр расписания своей группы на день/неделю. **[Must Have]**
*   Просмотр детальной информации о Занятии (преподаватель, аудитория/ссылка). **[Must Have]**
*   Просмотр списка доступных Очередей по своим предметам. **[Must Have]**
*   Запись на свободный Слот в Очереди. **[Must Have]**
*   Отмена своей записи в Очереди. **[Must Have]**
*   Просмотр своих записей в Очередях. **[Must Have]**
*   Написание заметок к предметам в расписании. **[Could have]**
*   Инициировать и пассивно участвовать в (пере-)выборах старосты. **[Could have]**
*   Отправить запрос старосте другой группы на добавление в очередь этой группы. **[Could have]**

**2.2.2. Староста**
*   Все возможности Студента. **[Must Have]**
*   Назначается админом либо получает роль в результате передачи роли "Староста" от другого старосты. **[Must Have]**
*   Управление очередями: создание, удаление, изменение ее параметров. **[Must Have]**
*   Передача роли Старосты. **[Should Have]**
*   Перенос неуспевших из старой очереди в новую. **[Should Have]**

**2.2.3. Администратор**
*   Все возможности Студента и Старосты. **[Must Have]**
*   Назначение старосты. **[Must Have]**
*   Управление пользователями: создание, блокировка, удаление аккаунтов, смена ролей. **[Must Have]**
*   Управление справочниками: создание и редактирование учебных групп, факультетов, предметов, аудиторий. **[Must Have]**
*   Просмотр системных логов. **[Must Have]**
Самый администратор системы - пользователь с заданными логином и паролем (заносится в БД). **[Must Have]**

### **2.3. Схема работы очереди**
Очередь имеет время создания, время открытия, время закрытия (опционально), размер (опционально). **[Must Have]**
Студенты могут записываться в очередь, созданную старостой. **[Must Have]**
Староста помечает после закрытия очереди, сколько человек успело пройти, и при формировании следующей очереди на этот предмет у него будет опция "Учесть неуспевших в последний раз", при этом неуспевшие будут автоматически перенесены в начало новой очереди, запись будет производиться после них. **[Should Have]**

### **3. Техническое описание**

#### **3.1. Архитектура решения**
Приложение будет реализовано по клиент-серверной архитектуре. Разделение приложения на три слоя по паттернам чистой архитектуры: репозиторий, бизнес-логика (usecase), контроллеры (хэндлеры).

*   **Клиент (Фронтенд):** Одностраничное приложение (SPA - Single Page Application), которое взаимодействует с сервером через REST API. Дизайн должен быть адаптивным для корректного отображения на десктопных и мобильных устройствах.
*   **Сервер (Бэкенд):** Предоставляет REST API для обработки запросов от клиента, реализует всю бизнес-логику и взаимодействует с базой данных. Аутентификация пользователей будет реализована на основе JWT (JSON Web Tokens).
*   **База данных (БД):** СУБД PostgreSQL для хранения структурированных данных о пользователях, расписании, очередях.

#### **3.2. Технологический стек**

*   **Фронтенд:**
    *   **Язык:** JavaScript
    *   **Фреймворк:** React
    *   **UI-библиотека:** Bootstrap
    *   **Сборщик:** Vite (или Webpack)
*   **Бэкенд:**
    *   **Язык:** Golang
    *   **Фреймворк:** Gin
*   **База данных:**
    *   **СУБД:** PostgreSQL
*   **Аутентификация**: JWT токены

### **5. Правила выгрузки (Deployment)**

#### **5.1. Окружения**
Проект будет иметь три окружения:
1.  **Development:** Локальные машины разработчиков.
2.  **Staging (тестовый стенд):** Сервер для предрелизного тестирования, полностью имитирующий продуктовое окружение. Деплой происходит автоматически при пуше в ветку `develop`.
3.  **Production (прод):** Основной сервер, доступный пользователям. Деплой происходит автоматически при мерже в ветку `main`.

#### **5.2. Контроль версий**
*   **Система:** Git.
*   **Хостинг репозитория:** GitHub.
*   **Модель ветвления:** GitFlow.
    *   `main` — стабильная, рабочая версия.
    *   `develop` — ветка для разработки.
    *   `feature/ticket_123_add_login_page` — ветки для разработки новой функциональности. Создаются от `develop`.

#### **5.3. CI/CD (Continuous Integration / Continuous Deployment)**
Будет настроен CI/CD-пайплайн с использованием GitHub Actions.
1.  **На push в `feature/*`:** Запуск линтеров и unit-тестов.
2.  **На merge в `develop`:** Запуск тестов, сборка Docker-образов, деплой на Staging-окружение.
3.  **На merge в `main`:** Запуск тестов, сборка Docker-образов, деплой на Production-окружение.

#### **5.4. Code Review**
Merge в ветку `develop` возможен только после одобрения минимум одним членом команды. Merge в ветку `main` возможен только после одобрения минимум двумя членами команды. При этом merge в любую ветку возможен только после прохождения пайплана проверок (состоящего как минимум из линтеров).
