# Техническое задание проекта

### **1. Описание артефактов и терминов (Глоссарий)**

| Термин | Описание |
| :--- | :--- |
| **Система** | Веб-приложение «Студ-Ассистент». |
| **Пользователь** | Общее название для любого лица, взаимодействующего с Системой. |
| **Студент** | Пользователь с ролью "Студент". Основной потребитель функциональности: просмотр расписания, запись в очереди, управление задачами. |
| **Администратор** | Пользователь с максимальными правами. Управляет всеми данными в системе: аккаунтами пользователей, группами, расписанием, настройками системы. |
| **Учебная группа** | Группа, в которой обучается Студент (например, "ИУ-81Б"). Используется для фильтрации расписания. |
| **Расписание** | Совокупность Занятий для учебных групп. Отображается в виде сетки по дням недели и времени. |
| **Занятие** | Элемент расписания, характеризующийся предметом, типом (лекция, практика), временем, местом (аудитория, ссылка на онлайн-конференцию) и преподавателем. |
| **Очередь** | Электронный список для записи Студентов на сдачу лабораторной работы, экзамена или консультации. |
| **Слот** | Конкретное время в Очереди, на которое может записаться один Студент. |
| **Задача** | Запись в персональном планировщике Студента. Имеет название, описание, срок выполнения и статус. |

### **2. Продуктовое описание**

#### **2.1. Цели и задачи продукта**
Создать единое информационное пространство для студентов, которое позволит:
*   Удобно отслеживать актуальное расписание занятий.
*   Цифровизировать и упорядочить процесс записи на сдачу работ и экзаменов.
*   Эффективно планировать личные и учебные задачи.

#### **2.2. Роли пользователей и их возможности**

**2.2.1. Студент**
*   Регистрация и авторизация в системе.
*   Просмотр и редактирование своего профиля (ФИО, группа).
*   Просмотр расписания своей группы на день/неделю.
*   Просмотр детальной информации о Занятии (преподаватель, аудитория/ссылка).
*   Просмотр списка доступных Очередей по своим предметам.
*   Запись на свободный Слот в Очереди.
*   Отмена своей записи в Очереди.
*   Просмотр своих записей в Очередях.
*   Создание, редактирование и удаление личных Задач в планировщике.
*   Изменение статуса Задачи (новая, в работе, выполнена).
*   Получение уведомлений (опционально в v1.0) о приближающемся занятии, записи в очередь, дедлайне задачи.

**2.2.2. Староста**
*   Все возможности Студента.
*   Управление очередями: создание, удаление, изменение параметров.
* Передача роли Старосты.

**2.2.3. Администратор**
*   Все возможности Студента и Старосты.
* Назначение старосты
*   Управление пользователями: создание, блокировка, удаление аккаунтов, смена ролей.
*   Управление справочниками: создание и редактирование учебных групп, факультетов, предметов, аудиторий.
*   Просмотр системных логов.

### **3. Техническое описание**

#### **3.1. Архитектура решения**
Приложение будет реализовано по клиент-серверной архитектуре.

*   **Клиент (Фронтенд):** Одностраничное приложение (SPA - Single Page Application), которое взаимодействует с сервером через REST API. Дизайн должен быть адаптивным для корректного отображения на десктопных и мобильных устройствах.
*   **Сервер (Бэкенд):** Предоставляет REST API для обработки запросов от клиента, реализует всю бизнес-логику и взаимодействует с базой данных. Аутентификация пользователей будет реализована на основе JWT (JSON Web Tokens).
*   **База данных (БД):** СУБД PostgreSQL для хранения структурированных данных о пользователях, расписании, очередях и задачах.

#### **3.2. Технологический стек**

*   **Фронтенд:**
    *   **Язык:** JavaScript
    *   **Фреймворк:** React
    *   **UI-библиотека:** Material-UI (или Ant Design / Bootstrap)
    *   **Сборщик:** Vite (или Webpack)
*   **Бэкенд:**
    *   **Язык:** Java
*   **База данных:**
    *   **СУБД:** PostgreSQL (или MySQL)
*   **Аутентификация:**
    *   **Механизм:** JWT (Access + Refresh токены)

#### **3.3. Структура API (основные эндпоинты)**

*   `POST /api/auth/register` - Регистрация нового пользователя.
*   `POST /api/auth/login` - Авторизация, получение токенов.
*   `POST /api/auth/refresh` - Обновление access-токена по действующему refresh-токену.
*   `POST /api/auth/logout` - Выход из системы (ревокация refresh-токена).
*   `GET /api/schedule?group_id=...&date=...` - Получение расписания для группы на указанную дату/неделю.
*   `GET /api/queues?subject_id=...` - Получение списка активных очередей по предмету.
*   `POST /api/queues/{id}/signup` - Запись текущего пользователя на слот в очереди.
*   `DELETE /api/queues/slots/{slot_id}/booking` - Отмена записи студента на слот (освобождение слота без его удаления).
*   `POST /api/teacher/queues` - Создание новой очереди.
*   `PUT /api/teacher/slots/{slot_id}` - Обновление статуса слота.
*   `GET /api/tasks` - Получение списка задач текущего пользователя.
*   `POST /api/tasks` - Создание новой задачи.
*   `PUT /api/tasks/{id}` - Редактирование задачи.
*   `DELETE /api/tasks/{id}` - Удаление задачи.

### **4. Условия эксплуатации**

#### **4.1. Требования к клиентской части**
*   **ОС:** Windows 10+, macOS 11+, современные дистрибутивы Linux, Android 9+, iOS 14+.
*   **Браузер:** Актуальные версии Google Chrome, Mozilla Firefox, Safari, Microsoft Edge.
*   **Требования:** Наличие постоянного подключения к сети Интернет.

#### **4.2. Требования к серверной части**
*   **ОС:** Linux (Ubuntu 22.04 LTS или новее).
*   **СУБД:** PostgreSQL 14 или новее.
*   **Контейнеризация:** Для развертывания используется Docker и Docker Compose.

### **5. Правила выгрузки (Deployment)**

#### **5.1. Окружения**
Проект будет иметь три окружения:
1.  **Development:** Локальные машины разработчиков.
2.  **Staging (тестовый стенд):** Сервер для предрелизного тестирования, полностью имитирующий продуктовое окружение. Деплой происходит автоматически при пуше в ветку `develop`.
3.  **Production (прод):** Основной сервер, доступный пользователям. Деплой происходит автоматически при мерже в ветку `main`.

#### **5.2. Контроль версий**
*   **Система:** Git.
*   **Хостинг репозитория:** GitHub.
*   **Модель ветвления:** GitFlow (упрощенный).
    *   `main` — стабильная, рабочая версия.
    *   `develop` — ветка для разработки.
    *   `feature/ticket_123_add_login_page` — ветки для разработки новой функциональности. Создаются от `develop`.

#### **5.3. CI/CD (Continuous Integration / Continuous Deployment)**
Будет настроен CI/CD-пайплайн с использованием GitHub Actions.
1.  **На push в `feature/*`:** Запуск линтеров и unit-тестов.
2.  **На merge в `develop`:** Запуск тестов, сборка Docker-образов, деплой на Staging-окружение.
3.  **На merge в `main`:** Запуск тестов, сборка Docker-образов, деплой на Production-окружение.
